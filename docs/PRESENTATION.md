# KTZ Digital Twin — Полное описание проекта

## Цифровой двойник локомотива: система мониторинга и визуализации телеметрии в реальном времени

---

## 1. Обзор проекта

KTZ Digital Twin — это полноценная микросервисная платформа для мониторинга состояния локомотивов казахстанской железной дороги в режиме реального времени. Система принимает поток телеметрических данных от подвижного состава, вычисляет индекс здоровья каждой подсистемы, генерирует интеллектуальные алерты, сохраняет историю для ретроспективного анализа и предоставляет оператору и диспетчеру интерактивные дашборды с визуализацией.

Проект охватывает полный цикл данных: от датчиков через потоковую обработку до интерактивной визуализации, включая экспорт отчётов, управление пользователями и развёртывание через Docker.

## 2. Архитектура системы

### 2.1. Микросервисная топология

```
                    │  (Единый UI)    │  Vite + Tailwind CSS

┌──────────────────┐      ┌──────────────────┐
│  back_locomotive │─Kafka│  back_dispatcher │
│  (Телеметрия)    │─────→│  (Агрегация)     │
│  FastAPI + Python│      │  FastAPI + Python │
└──────────────────┘      └────────┬─────────┘
                                   │
                          ┌────────▼─────────┐
                          │   TimescaleDB    │
                          │   (PostgreSQL)   │
                          └──────────────────┘
```

### 2.3. Транспортный протокол

Все WebSocket-сообщения передаются в стандартизованном конверте:

```json
{
  "type": "telemetry.frame",
  "payload": { ... },
  "timestamp": 1710000000000,
  "sequenceId": 42,
  "event": {
    "event_id": "uuid",
    "event_type": "telemetry.frame",
    "source": "back_locomotive",
    "locomotive_id": "KTZ-2001",
    "occurred_at": 1710000000000,
    "schema_version": "1.0"
  }
}
```

Контракт событий (Event Envelope V1) обеспечивает:
- **Дедупликацию** — по `event_id` (LRU-буфер на 20 000 последних ID)
- **Трассировку** — каждое событие идентифицируемо от источника до потребителя
- **Валидацию** — проверка `schema_version`, соответствие `event_type` и `type`, проверка `locomotive_id`
- **Упорядоченность** — монотонный `sequenceId` на процесс

## 3. Backend: Локомотивный сервер (`back_locomotive`)

### 3.1. Физический симулятор телеметрии

Сердце проекта — физически корректный симулятор, моделирующий поведение тепловоза TE33A:

- **4-фазный цикл движения**: idle → acceleration → cruise → braking, с реалистичными длительностями и переходами
- **14 телеметрических метрик** в 5 группах, каждая с физическими зависимостями
- **EWM-сглаживание** (Exponentially Weighted Moving Average) для предотвращения скачков
- **Физически корректные зависимости**: скорость влияет на тяговый ток, ток — на температуру, торможение — на давление в магистрали
- **Профиль уклона** — синусоидальная модель рельефа влияет на нагрузку

#### Метрики и их физические связи:

| Группа | Метрики | Физическая модель |
|--------|---------|-------------------|
| **Движение** | Скорость (км/ч), Ускорение (м/с²), Расстояние (км) | Фазовый автомат с EWM-демпфированием |
| **Топливо** | Уровень (%), Расход (л/ч) | Расход = f(скорость, нагрузка), бак 5000 л |
| **Температура** | Охлаждающая жидкость (°C), Масло (°C), Выхлоп (°C) | Трёхслойная тепловая EWM-модель |
| **Давление** | Главный резервуар (бар), Тормозная магистраль (бар), Масло (бар) | Тормоза зависят от команд торможения, масло — от оборотов |
| **Электрика** | Тяговое напряжение (В), Тяговый ток (А), Батарея (В) | Ток = f(ускорение, уклон), напряжение = f(обороты, нагрузка) |

### 3.2. Подсистема паттернов неисправностей

Система включает **10 предопределённых профилей неисправностей**, каждый моделирующий конкретный сценарий отказа:

| ID | Неисправность | Ключевые отклонения |
|----|--------------|---------------------|
| KTZ-BRK-001 | Неисправное торможение | Высокая скорость при жёстком тормозе, слабый отклик давления |
| KTZ-PNE-002 | Потеря пневматического давления | Давления ниже безопасного уровня |
| KTZ-OIL-003 | Масляное голодание | Критически низкое давление масла, перегрев |
| KTZ-THM-004 | Тепловой разгон | Все температуры выше критического диапазона |
| KTZ-DRV-005 | Высокий ток / слабое ускорение | Ток 980 А, ускорение 0.01 м/с² |
| KTZ-VLT-006 | Просадка напряжения под нагрузкой | Напряжение 2460 В при токе 820 А |
| KTZ-AMP-007 | Перегрузка по тяговому току | Ток 1710 А (выше warningHigh) |
| KTZ-FUL-008 | Критически низкий запас топлива | Начальный уровень 7.8%, ускоренный расход |
| KTZ-BRN-009 | Неэффективный расход | Расход 182 л/ч на скорости 12 км/ч |
| KTZ-MIX-010 | Комбинированная неисправность | Тормоза + масло + просадка напряжения |

Каждый профиль генерирует детерминированную, но живую телеметрию с синусоидальными микроколебаниями. Это позволяет демонстрировать одновременную работу 10+ локомотивов с различными сценариями.

### 3.3. Движок индекса здоровья

Индекс здоровья (Health Index) рассчитывается по каждой из **6 подсистем**:

- **Двигатель** (engine) — масло, температуры, выхлоп
- **Тормоза** (brakes) — давления тормозной магистрали и главного резервуара
- **Электрика** (electrical) — напряжение, ток, батарея
- **Топливо** (fuel) — уровень, расход
- **Охлаждение** (cooling) — температура охлаждающей жидкости
- **Пневматика** (pneumatic) — давление в главном резервуаре

**Алгоритм на стороне `back_locomotive`:**
1. Случайный дрейф (±0.5 по шкале 0–100) для реалистичности
2. Пороговые штрафы: warning = -5, critical = -15 очков
3. Clamping к [0, 100]
4. Маппинг score → status: ≥80 normal, ≥60 degraded, ≥40 warning, <40 critical
5. Подсчёт активных алертов на подсистему
6. Overall = среднее всех подсистем

**Ключевая фича — Explainability (объяснимость):**

Каждый штраф записывается как `SubsystemPenalty` с полями:
- `metricId`, `metricLabel` — что именно вышло за порог
- `currentValue` — текущее значение
- `thresholdType` — какой порог нарушен (warningHigh, criticalLow, etc.)
- `thresholdValue` — числовое значение порога
- `penaltyPoints` — сколько очков снято

Top-5 штрафов выносятся в `HealthIndex.topFactors` — оператор видит конкретные причины ухудшения здоровья, а не просто число.

### 3.4. Kafka Producer

Локомотивный бэкенд публикует каждый телеметрический фрейм, обновление здоровья и все события в Apache Kafka:

- Автоматическое создание топика при старте (с поддержкой расширения партиций)
- Идемпотентный producer (`enable_idempotence=True`, `acks="all"`)
- Партицирование по `locomotive_id` — гарантия порядка событий для каждого локомотива
- Валидация Event Envelope перед публикацией

### 3.6. WebSocket Broadcasting

6 фоновых задач:
1. **Телеметрия** — агрегация raw-сэмплов в фрейм каждую секунду
2. **Здоровье** — пересчёт индекса каждые 5 секунд
3. **Heartbeat** — проверка связи каждые 10 секунд
4. **Алерты** — генерация ситуативных алертов
5. **Сообщения** — диспетчерские сообщения
6. **Pattern Fleet** — параллельная генерация фреймов для всех 10 паттернов неисправностей

### 3.7. Конфигурируемые пороги

Все пороги метрик и штрафы хранятся в `shared/thresholds.json` и монтируются в контейнеры. Диспетчер может обновить пороги через `PUT /api/config/thresholds` — без перезапуска и перекомпиляции. Изменения применяются мгновенно к движку здоровья.

### 3.8. Аутентификация

- REST: заголовок `X-API-Key`
- WebSocket: query-параметр `?apiKey=...` или JWT-токен `?token=...`
- `/ping` остаётся публичным

## 4. Backend: Диспетчерский сервер (`back_dispatcher`)

### 4.1. Двойной режим инжеста

Диспетчер поддерживает **три режима приёма данных** (`INGEST_MODE`):
- `ws` — подключение к каждому локомотивному WebSocket
- `kafka` — потребление из Kafka-топика
- `hybrid` — оба канала одновременно

В Kafka-режиме:
- Валидация каждого конверта (schema_version, event_type match, locomotive_id)
- Мониторинг consumer lag (логирование каждые 10 секунд)
- Автовосстановление при обрыве соединения

В WS-режиме:
- Outbound WebSocket-клиенты к каждому локомотиву
- Экспоненциальный backoff при реконнекте
- Ping/pong keepalive

### 4.2. Интеллектуальный Rule Engine (Health Engine)

Движок здоровья на диспетчерской стороне значительно сложнее локомотивного — он использует **временные окна и мультиметрические корреляции**:

**Временные окна:** 1с, 5с, 30с, 10 мин — для каждого окна вычисляются средние значения ключевых метрик.

**Составные правила алертов:**

| Алерт | Правило | Серьёзность |
|-------|---------|-------------|
| Слабый отклик тормозов | speed > 50 И hard_brake И brake_main < 7.0 | Critical |
| Нет замедления при торможении | speed > 60 И hard_brake И speed_drop_3s < 5 | Critical |
| Потеря пневмодавления | speed > 50 И brake_pipe < 4.3 И brake_main < 7.0 | Critical |
| Критически низкое давление масла | oil_pressure < criticalLow И speed > 20 | Critical |
| Тепловая перегрузка | coolant > criticalHigh ИЛИ oil > criticalHigh ИЛИ exhaust > criticalHigh | Critical |
| Ухудшение торможения | speed > 40 И brake_command И brake_main < 7.2 | Warning |
| Высокий ток / слабое ускорение | current_5s > 650 И accel < 0.05 И not_braking И speed < 70 | Warning |
| Просадка напряжения под нагрузкой | voltage < warningLow И current > 400 | Warning |
| Тенденция пневмодавления | brake_pipe < 4.8 И speed > 20 | Info |
| Повышенный тяговый ток | current_5s > warningHigh | Info |
| Низкий запас топлива | fuel_level < warningLow | Info |

**Взвешенный Health Score:**
```
overall = 0.40 × brakes + 0.25 × thermal + 0.20 × powertrain + 0.15 × fault
```

**Жизненный цикл алертов:**
- `alert.new` — при первом срабатывании условия
- `alert.update` — при изменении severity или описания
- `alert.resolved` — когда условие перестаёт выполняться

### 4.3. TimescaleDB: персистенция и Replay

Все входящие данные сохраняются в TimescaleDB:

| Таблица | Назначение |
|---------|-----------|
| `telemetry_points` | Временные ряды метрик (locomotive_id, metric_id, ts, value) |
| `health_snapshots` | Снимки индекса здоровья |
| `alert_events` | Полный лог жизненного цикла алертов |
| `incoming_messages` | Сообщения от локомотивов |
| `dispatcher_commands` | Команды от диспетчера |

**Миграции** через Alembic (5 миграций): создание схемы, обновление replay, аутентификация, логирование, нормализация ролей.

**Replay API:**
- `GET /api/locomotives/{id}/replay/time-range` — диапазон доступных данных
- `GET /api/locomotives/{id}/replay/range?from=...&to=...&resolution=...` — история с агрегацией
- `GET /api/locomotives/{id}/replay/snapshot?timestamp=...` — полный снимок состояния в момент времени

**Разрешения Replay:**
- `raw` — каждый сэмпл
- `1s`, `10s`, `1m`, `5m` — агрегированные бакеты (SQL GROUP BY с floor)

**Snapshot** — восстанавливает полное состояние на момент времени:
1. Последние значения каждой метрики до выбранного timestamp (row_number window function)
2. Ближайший health snapshot
3. Полный лог alert events с реконструкцией active/resolved

### 4.4. Backpressure и Fan-Out

Dispatcher использует продвинутую систему доставки сообщений клиентам:

- **Per-client runtime** (`DispatcherClientRuntime`): отдельная очередь, sender task, счётчики потерь
- **Телеметрия — latest-wins**: если в очереди уже есть фрейм для данного локомотива, новый заменяет старый
- **Health — latest-wins** аналогично
- **Алерты и сообщения — ordered queue**: гарантия доставки и порядка
- **Метрики backpressure**: `telemetry_drop_count`, `health_drop_count`, `send_failure_count`, `queue_depth`
- **Subscription filtering**: клиент подписывается на конкретный `locomotive_id` или `"*"` — диспетчер отправляет только релевантные данные

### 4.5. Полная аутентификация и авторизация

**JWT-based auth** с refresh-токенами:

| Роль | Возможности |
|------|------------|
| **admin** | Полный доступ, управление пользователями, конфигурация |
| **dispatcher** | Мониторинг всех локомотивов, диспетчерский чат |
| **regular_train** | Только свой локомотив, ограниченный UI |

- Пароли хешируются **Argon2** (winner Password Hashing Competition)
- JWT-токены с HMAC подписью
- Refresh token в httpOnly cookie
- Принудительная смена пароля при первом входе
- Audit log в БД (login, password_change, user management)
- Автоматический seed demo-пользователей при старте

### 4.6. Диспетчерский чат

Двусторонняя связь диспетчер ↔ машинист:
- Диспетчер отправляет `dispatcher.chat` через WebSocket
- Сообщение пересылается на WebSocket целевого локомотива
- Сохраняется в TimescaleDB (`dispatcher_commands`)
- Broadcast всем подписанным клиентам как `message.new`

### 4.7. Runtime Stats и мониторинг

Эндпоинт `GET /api/health` возвращает полную диагностику:
- Время работы, количество подключений/отключений, пиковое число клиентов
- Общее количество ingested/broadcast сообщений по типам
- Per-client: глубина очереди, количество дропов, причины отключений

## 5. Frontend: Единый интерфейс (`front_locomotive`)

### 5.1. Архитектура фронтенда

- **React 19** с TypeScript — строгая типизация всего API контракта
- **Zustand 5** — 10 глобальных стейт-сторов:
  - `useTelemetryStore` — текущие и сглаженные показания, sparkline-буферы
  - `useHealthStore` — индекс здоровья по локомотивам
  - `useAlertStore` — жизненный цикл алертов (new/update/resolved)
  - `useMessageStore` — диспетчерские сообщения
  - `useConnectionStore` — состояние WebSocket, heartbeat, реконнект
  - `useFleetStore` — реестр локомотивов и их сводка
  - `useReplayStore` — состояние воспроизведения истории
  - `useSettingsStore` — пользовательские настройки (сглаживание, альфа)
  - `useDispatchConsoleStore` — чат-история диспетчерской консоли
  - `useAuthStore` — сессия, токены, роль пользователя

- **TanStack Query 5** — кэширование REST-запросов с автообновлением
- **5 адаптеров** (`services/adapters/`) — нормализация snake_case/camelCase между бэкендом и фронтендом
- **MSW (Mock Service Worker)** — полный набор mock-handlers для автономной разработки без бэкенда

### 5.2. Страница: Dashboard (Панель состояния парка)

Главная страница показывает **все подключённые локомотивы** в виде карточек:

- **Fleet Health Cards** — для каждого локомотива: Health Gauge, скорость, топливо, охлаждение, количество алертов, статус подключения, время последнего обновления
- **Locomotive Detail Modal** — при клике на карточку открывается детальный вид:
  - **Health Gauge** — анимированный ECharts gauge (0–100) с цветовой кодировкой
  - **Subsystem Bars** — индексы здоровья 6 подсистем
  - **Health Explainer** — Top-5 Contributing Factors с конкретными метриками и штрафами
  - **Alert Feed** — последние 5 алертов
  - **Live Metrics** — карточки метрик по группам с sparkline-графиками
  - **Dispatcher Inbox** — последние сообщения

- **Export Menu** — экспорт CSV телеметрии, CSV алертов, печатный PDF-отчёт

- **Ролевая фильтрация** — пользователь `regular_train` видит только свой локомотив

### 5.3. Страница: Телеметрия

Детальная страница мониторинга метрик:

- **DynamicMetricRenderer** — адаптивные карточки метрик с числовым значением, единицей измерения и sparkline-графиком
- **EMA-сглаживание** — переключаемое сглаживание (alpha = 0.3) для устранения визуального дрожания без потери raw-данных
- **LineChart с dataZoom** — интерактивные графики:
  - Scroll zoom (`type: "inside"`)
  - Slider zoom (`type: "slider"`) с кастомной стилизацией
  - User zoom state tracking — ручной зум не сбрасывается при обновлении данных
- **TimeRangeSelector** — пресетные окна: 1 мин, 5 мин, 15 мин, 1 ч, Все
- **Trend Buffers** — раздельные буферы для sparkline (60 точек) и trend (3600 точек)
- **CSV Export** — скачивание через Streaming Response

### 5.4. Страница: Схема локомотива (Diagram)

**Интерактивная 2D SVG-схема тепловоза TE33A** с 8 зонами:

### 5.5. Страница: Карта (Map)

**Leaflet-карта Казахстана** с симуляцией 10 поездов:

- **OpenStreetMap** базовый слой + **OpenRailwayMap** слой железных дорог (переключаемый)
- **Rail-aligned routing** — маршрут привязывается к реальным железнодорожным путям:
  1. Базовый fallback-маршрут по коридору Казахстана
  2. Запрос к Overpass API для получения реальной геометрии ж/д путей
  3. Алгоритм buildRailAlignedRoute выравнивает маршрут по ближайшим рельсам
- **10 поездов** — каждый с уникальным цветом, скоростью и позицией
- **Симуляция движения** — тик каждую секунду, скорость с синусоидальной осцилляцией
- **Trail** — полилиния следа за каждым поездом (до 180 точек)
- **Popup** — при клике: ID, скорость, координаты

- **AlertChip** — компактное отображение с иконкой severity
- **Alert Feed** — лента с сортировкой по серьёзности и времени
- **Acknowledge** — подтверждение алерта оператором (POST request)

### 5.7. Страница: Сообщения (Messages)

- Лента диспетчерских сообщений

### 5.8. Страница: Replay (История)

Полнофункциональный плеер исторических данных из TimescaleDB:

- **TimelineScrubber** — полноширинная полоса прокрутки по временному диапазону
- **PlaybackControls** — Play/Pause, скорость (1x/2x/5x/10x), skip ±10с
- **ReplayChart** — LineChart с вертикальным маркером текущего момента и threshold reference areas
- **ReplayMetricSelector** — чекбоксы метрик для выбора отображаемых графиков
- **ReplaySnapshotSummary** — снимок здоровья, алертов и метрик в выбранный момент времени
- **Resolution control** — raw, 1s, 10s, 1m, 5m (серверная агрегация)
- **Time range presets** — 1 мин, 5 мин, 15 мин
- **Export** — CSV и печатный PDF с разбивкой по подсистемам, штрафам и алертам

### 5.9. Страница: Диспетчерская консоль (Dispatch Console)

Доступна ролям `admin` и `dispatcher`:

- **Список локомотивов** — подключённые локомотивы с краткой сводкой

### 5.10. Страница: Управление пользователями (User Management)

Доступна только роли `admin`:

### 5.11. Аутентификация

## 6. Потоковая обработка: Apache Kafka

### 6.1. Конфигурация

- **KRaft mode** — без ZooKeeper (Kafka 3.x+)
- Топик `ktz-events` (конфигурируемый)
- Настраиваемое количество партиций и replication factor
- Consumer group `ktz-dispatcher`
- Auto-offset: `latest` — диспетчер подключается к живому потоку

### 6.2. Гарантии доставки

- **Producer**: `acks=all`, `enable_idempotence=True` — exactly-once semantics на уровне producer
- **Consumer**: валидация envelope, дедупликация по event_id
- **Ordering**: партицирование по `locomotive_id` — гарантия порядка для каждого локомотива

## 7. Хранение данных: TimescaleDB

- **PostgreSQL 16** с расширением TimescaleDB 2.15
- Оптимизировано для временных рядов телеметрии
- Автоочистка (`prune_telemetry`, `prune_health_snapshots`) по настраиваемому retention
- Индексы для эффективного запроса по `locomotive_id + ts`
- Конфликт-резолюция через `ON CONFLICT DO NOTHING` по `source_event_id`

## 8. Docker Compose: Развёртывание одной командой

```bash
./scripts/start_microservices.sh up
```

Запускает 5 сервисов:
1. **Kafka** (KRaft, single-node) — порт 9092
2. **TimescaleDB** — порт 5433, с healthcheck
3. **back_locomotive** — порт 3001, зависит от Kafka
4. **back_dispatcher** — порт 3010, зависит от Kafka + TimescaleDB (healthy)
5. **front_locomotive** — порт 5183, многоэтапная сборка (Node build → Nginx serve)

**Shared volume** — `shared/thresholds.json` монтируется в оба бэкенда для единой конфигурации порогов.

**Environment** — все настройки через `.env.microservices`:
- URL-ы, порты, ключи
- Kafka bootstrap, topic, partitions
- Database URL, retention
- Auth: token secret, TTL, demo users
- Инжест-режим (ws/kafka/hybrid)

## 10. Безопасность

| Механизм | Реализация |
|----------|-----------|
| API-key auth | Все REST/WS запросы требуют ключ (кроме `/ping` и auth endpoints) |
| JWT tokens | Access + Refresh, HMAC-подпись, настраиваемый TTL |
| Password hashing | Argon2 (победитель Password Hashing Competition) |
| RBAC | 4 роли: admin, dispatcher, regular_train, service |
| Locomotive isolation | regular_train видит только свой locomotive_id |
| Force password change | При первом входе или сбросе пароля |
| Audit logging | Все auth-события записываются в БД |
| CORS | Настраиваемый список разрешённых origins |
| WebSocket auth | JWT token или API key через query params |
